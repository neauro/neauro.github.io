Remote Procedure Calls (RPC)

Jan 4th, 2011

What is a remote procedure call (RPC)?
RPC is a paradigm for providing communication across a network between programs written in a high-level language. “Procedure calls” in general are a well-known and well-understood way of transferring control and data within a program on a single computer – a program calls a procedure, giving it certain arguments, and receives some output from the procedure back. So, a remote procedure call – when a program calls a procedure that is not necessarily on the same system that the program is on – should be workable as well.

An RPC “cycle” goes like this:
  1. A remote procedure is invoked by a caller, who passes the procedure parameters
  2. The caller’s environment is suspended as it waits for a response
  3. The parameters are passed across the network to the environment where the procedure will execute
  4. The desired procedure is executed
  5. When the procedure finishes executing, the results are passed back to the calling environment
  6. The caller’s environment resumes with the new data (or else fails, I assume, because of a timeout…though the “Implemented remote procedure calls” didn’t use timeouts, since single-machine systems don’t)


What are the advantages of RPC?
  1. Clean and simple semantics. Distributed systems are complicated because communication throughout one is complicated, so simplicity is good.
  2. Simplicity. Which means it’s probably fast.
  3. Generality. Procedure calls work in single-machine systems, besides being usable in distributed systems.


What are the disadvantages of RPC?
There are some design issues –
  • What semantics do you use when the machine fails, or when communication fails?
  • What are the semantics when a caller does something address-specific, and the caller and callee don’t have a shared address space?
  • How do you integrate RPC into existing/future programming systems?
  • How does a caller figure out the location and identity of the procedure it wants?
  • What’s a suitable protocol for the transfer of data?
  • How do you provide data integrity and security in an open communication network?


Really general implementation
The structure of RPC is based on the concept of stubs; the parts of the program involved in an RPC (the user, the user-stub, the RPC communications package, the server-stub, and the server) are all executed on their respective places, and when a user wants to use an RPC, it really just calls its user-stub, which is what makes the RPC. This way, specifying the target procedure and the arguments is sent in a “call packet” to the server-side stuff, which is handled similarly: the server-stub unpacks the packet, makes a local call to the procedure on the server, and packs it up again in a “result packet” to be sent to the caller. This serves to encapsulate everything (I think?).

from “Implementing Remote Procedure Calls”

The RPCRuntime handles the actual handling-off between the user-stub and the server-stub.

The semantics of RPCs can also be handled without needing a specialized packet-level protocol.


Other notes:
  • RPC is supposed to make distributed computation more easy; in the “Implementing Remote Procedure Calls” paper, the authors hoped to solve the difficulty of distributed systems communication and leave only the problems of timing, independent failure of components, and the coexistence of independent execution environments.


Source:
Andrew Birrell and Bruce Nelson. Implementing Remote Procedure Calls. ACM TOCS 1984.

Intro to Distributed Systems

Jan 4th, 2011

So, for now at least, I’m taking a class on distributed systems. Google has a good tutorial on what a distributed system is, which I will take notes from/copy shamelessly.

Firstly, definitions –
A protocol is “a formal description of message formats and the rules that two processes must follow in order to exchange those messages.”

A network is “the infrastructure that links computers, workstations, terminals, servers, etc. It consists of routers which are connected by communiation links.”

A distributed system is “an application that executes a collection of protocols to coordinate the actions of multiple processes on a network, such that all components cooperate together to form a single or small set of related tasks.”

My impression of it currently is also like this: a distributed system is a program that controls a ton of computers and with their powers combined accomplishes great things, like searching the internets for that picture of Maru wearing a crab hat.


Advantages of a distributed system:
Distributed systems can connect remote users with remote resources in an open and scalable way. Open meaning that each component of the system can interact with other components at any time, scalable meaning that the system can be easily altered to accomodate changes in the amount of users or resources.


Disadvantages of a distributed system:
A system of things working together is more complicated than a system of something working by itself. An example of this is an airplane with two engines, which spend much more time on the ground than airplanes with a single engine, simply because planes cannot take off if anything is wrong with them, and there’s more to go wrong with an airplane that has more engines.

Distributed systems have a lot of complexity because its components, the computers or servers or etc., have to interact properly. An example of how difficuult the interaction can be can be found with the two generals problem, which goes like this:

Two generals on opposite hillsides want to coordinate an attack. General A sends a message – “Let’s attack at dawn!” – to General B. How will General A know if General B agrees? B will send a message in response – “Alright, let’s attack at dawn!” But then how will B know that A received his confirmation? A will have to send a message back. And then how will A know that B got the confirmation message? B will have to send a message back…

And so on and so forth.


So, a well-designed distributed system will have these characteristics:
  1. Fault-tolerant: if a components fails, it can recover without performing incorrect actions
  2. Highly available: if a component fails, it can restore operations and continue to provide services
  3. Recoverable: if a component fails, it can restart itself and rejoin the system after failure has been fixed
  4. Consistent: in the event of failure, a system can still coordinate components
  5. Scalable: if some aspect of the system is scaled to a larger size (i.e. if there are more users, or size of network increases), then the system can still work correctly
  6. Predictable performance: the system can provide responsiveness in a timely fashion
  7. Secure: the system can authenticate access to data and services

With distributed systems, you have to design for failure. If I told you to do something and you didn’t do it, is it because I didn’t tell you clearly what to do? Or is it because you didn’t see me tell you? Or is it because I didn’t get confirmation that you had done it? This problem is the reason why distributed systems are so hard to debug – it’s hard to see where exactly the problem originated, or why it happened.


Distributed systems design
Firstly, about client-server applications. They work by having a server which provides some service, like processing database queries, and a client which uses the service, i.e. to show the database query results. Different types of servers do different things: for instance, a database server houses databases and allows clients to access them, file servers manage disk storage, etc. The communication between client and server needs to be reliable.

Reliability is governed by protocols, like TCP/IP. The Internet Protocol (IP) is a set of communication protocols that allow for communication on the internet and most commercial networks. The Transmission Control Protocol (TCP) is a core part of IP, and allows clients and servers to create connections to one another and exchange data in “packets.” Protocol guarantees reliable and in-order delivery of data from sender to receiver.

With a distributed system, there can be many types of servers, and service is a term that refers to a set of servers of a particular type. “Binding” is when a process that needs to access a service becomes associated with a particular server which provides the service. Programs which which server to associate with through binding policies. For example, someone looking for certain information on a database may be connected to a server which is closest to them geographically, so that access is faster, or he or she may be connected to a database which has the least number of people presently accessing it.

To keep things reliable, a distributed service may keep multiple copies of data so you can still access it if something goes wrong. To keep things going fast, a distributed service may cache data, which is when it makes copies of the data and keeps a copy of it locally, so the data can be accessed quickly the next time it’s wanted.


More about TCP/IP
The IP “suite” can be viewed as a set of layers, in which each layer uses the functions of the layers below it and gives functionality to the layer above it. The lower layers are typically implemented in hardware, whereas the higher layers are implemented in software.

The layers:
  1. Application layer: is used by most programs that require netwrok communication. The program gives data to this layer of protocol, and then encapsulated in a transport layer protocol. Examples of applications that use this layer: HTTP, FTP.
  2. Transport layer: is responsible for end-to-end message transfer, along with error control, fragmentation, and flow control. End-to-end message transmission can be categorized as connection-oriented (TCP) or connectionless (UDP), with TCP being the more sophisticated and reliable.
    • TCP works by: first, ensuring the receiving computer is ready to accept data by using a “three-packet handshake” in which the sender and receiver agree they’re ready to communicate. Then, TCP makes sure data gets to its destination. If the receiver doesn’t acknowledge a packet of the data, TCP will automatically transmit the packet three times, and also split the large packet into smaller ones if needed so that the data can travel more reliably. TCP will also drop duplicate packets and rearrange the ones that arrive out of sequence.

Also, I understand that TCP is a protocol, which means that it’s a “formal description,” so what’s with this “TCP making sure data reaches” and “TCP automatically transmitting” stuff as if it’s a program which does something?
  3. Network layer: solves the problem of getting packets across a single network, or, getting packets from one network to another network by moving the packet across a network of networks, i.e. the Internets. IP performs the basic task of getting packets of data from source to destination.
  4. Link layer: deals with the physical transmission of data, i.e. placing frame headers and trailers on packets for travelling over the physical network, and dealing with physical components.


Remote Procedure Calls (RPCs)
A remote procedure call (RPC) is another method besides TCP/IP that clients can use to interact with servers. It’s based off the idea that the procedure that a client wants to use may not be in the same “address space,” i.e. the procedure is off on a different system with a network connecting them.

So, when an RCP is made, the arguments are passed to a remote procedure and the caller waits for a response or a timeout. Each incoming request to a server usually spawns a new thread, and a thread in the client issues an RPC and then waits, and resumes execution when it receives a reply.

RPC-based code needs to do three things:
  1. Specify the protocol for client-server communication
    • The protocol is created by stubs, which are generated by a protocol compiler. A stub is a routine that doesn’t do much other than to declare itself and the parameters it accepts – just enough to allow it to be compiled and linked.
  2. Develop the client program, which will call the remote procedure, pass it the required data, and receive returned data
  3. Develop the server program, which will register the procedures that may be called by a client, and receive/return data required for processing.

So, an RPC application uses classes made by the stub generator to execute an RPC, and then wait for it to finish.

Some errors that may happen in RPC programming are binding errors, which will happen when a server isn’t running when a client is started); or version mismatches, if a client compiled against an outdated version of the server. Timeouts will occur in the case of a server crash, network problem, or client computer problem.

Ingredients
  • 1 cup (2 sticks) unsalted butter, room temperature
  • 1 3/4 cups sugar
  • 2 large eggs
  • 2 large egg whites
  • 3 cups all-purpose flour
  • 2 teaspoons baking powder
  • 1/8 teaspoon salt
  • 1 cup milk
  • 2 tablespoons matcha

Instructions
  1. Preheat oven to 350 degrees.
  2. If you’ve got your roommate’s plastic-rubber cupcake things, you’re all set. Else, prepare paper cupcake things by spraying with butter stuff…I guess. (This recipe apparently also makes a cake, so you could butter up a 1/2 pan.)

Sit down on the ground, set the bowl between your thighs, and beat furiously. You could think about that ex-whatever that you had (but avoid thinking about how green tea is his favorite flavor…oh). At the end, the butter should be kind of whippy.
  4. Add sugar. Repeat the between-legs beating thing. The mixture should be light and fluffy and could possibly be on your pants and the cuff of your shirt.
  5. Beat two eggs with a fork and add it to the mixture. Walk around the kitchen if you want. Skip some songs on your iPod until you have something sort of fast-paced. (Last Alliance’s “Katahiza no Yogore” is good.) By now your arm might be getting kind of tired. Don’t worry. Just keep thinking about green tea cupcakes.
  6. Separate the whites from another two eggs, beat the whites with a fork, and add, and mix until…everything’s all mixed together. Optional: You might accidentally allow the yolk to bleed into the whites. This is (probably) okay. Add it anyway. Extra note: The initial recipe actually called for two yolks, but I read it as “whites” somehow, so…well…by the time of this writing I’ve actually made this recipe a couple of times, and it worked both ways, pretty much.   7. Whisk Grab another fork and another bowl and mix together flour, baking powder, and salt in a bowl. Add to batter and mix to combine.
  8. Mix matcha in with the milk. Add to the batter and mix until combine.
  9. Pour into cupcake things, about 3/4 of the way up.
  10. Bake for 22-25 minutes until a cake tester comes out clean.
  11. Remove from baking pans and then put something on them. My favorite topping is cream cheese icing, but this time around we only put more green tea, which had the unfortunate side-effect of making the cupcakes look ominously moldy.
  12. Set on a plate and offer to your roommates. Take a bunch of pictures, pour out some sake, and consume~

Developing Protocol Buffers

Jan 4th, 2011

Again taken from t3h googs, who developed protocol buffers in the first place.


What are protocol buffers?
Protocol buffers are a “flexible, efficient, automated mechanism for serializing structured data” – like XML but smaller, faster, simpler. In this case, I assume that “serializing data” refers to converting some data structure into an object that can be stored and then unpacked again into another computer environment, i.e. storing a list into a file or something so it can be sent over a network and then opened up again and read by someone else. So, a protocol buffer provides a way for data to be moved across multiple computers, which explains why it’s relevant to distributed systems.


How do protocol buffers work?
You have data, and you want to serialize it so that you can trasmit it to another network or computer in a message. To specify how you want to serialize it, you have to define protocol buffer message types in .proto files. Each protocol buffer message is a small, logical record of information with a series of name-value pairs. A .proto file which defines a message containing information about a Pokemon would be like this:

message Pokemon {
    required string species = 1;
    required int32 id = 2;
    optional string trainer = 3;
    optional string nickname = 4;

    enum Type1 {
        FIRE = 0;
        WATER = 1;
        GRASS = 2;
    }
}

So in this case, I’ve got a message that serializes data about a Pokemon. The message contains information about a Pokemon, its trainer, its id, its nickname, and its type. For a message to be a well-formed Pokemon message, it must contain exactly one of species and id, but can have zero or exactly one of trainer or nickname. The enum means that the Pokemon will have only one of a predefined list of values – that is, only one type out of the three given types.

Once the messages have been defined, they’re run against whatever protocol buffer compiler you have for your application’s language, which will generate data access classes. These will provide a way to access each field defined in the message. So, if you were using C++, the data access class created would be called Pokemon, and you could then use this Pokemon class to populate, serialize, and retrieve Pokemon protocol buffer messages. Then you send out the message like,

Pokemon pokemon;
pokemon.set_species("Charmander");
pokemon.set_id(4);
pokemon.set_trainer("Red");
fstream output("file", ios::out | ios::binary);
pokemon.SerializeToOsteam(&output);

and read it like,

fstream input("file", ios::in | ios:: binary);
Pokemon pokemon;
pokemon.ParseFromIsteam(&input);
cout << "Pokemon: " << pokemon.species() << endl;
cout << "Trainer: " << pokemon.trainer() << endl;

What are the advantages of protocol buffers?
  • You can add new fields to message formats without breaking backwards-compatibility; old binaries will just ignore the new field. So, extending the protocol is simple, and you can do it without having to worry about breaking existing code.
  • It’s simpler, smaller, faster, and less ambiguous than XML. Plus, it generates the data access classes, which are easier to use in programs. I imagine that XML takes so much space and is harder to parse because you have to close tags. Also, to manipulate XML you’d have to do things like finding a node “species” and then accessing its inner text, instead of just retrieving “species” immediately.

Problem: How Do I Use Flex Components to Make UI for an ActionScript Project?

Dec 30th, 2010

I’ve got an ActionScript project which already compiles into a SWF file of 600x400. So how do I add UI (buttons, etc.) to this? i.e. maybe import all the existing AS code into a Flex Project so I can use the fancy UI components?

I near-immediately discovered the SWFLoader, which had a relatively promising name. It’s a control that apparently “lets you load one application into another application.” You add the SWFLoader component, set its Source, and Flex will import the SWF file and run it. SWFLoader can also load different types of images (jpg, png, etc.) but I think that the Image component is supposed to be better suited for that.

Concerning design: an SWF that is loaded with an SWFLoader could be either a sub-application, or a module. The difference is that a main application would load a sub-application, instead of embedding their functionality, which would mean a smaller initial download size and shorter load time for the main application. This also means that you have better encapsulation of the functionality.

In an application that is modular, the main application would load each form as a module using the ModuleManager, which maps URLs to modules, and dispatches events to communicate whether a module is in a loaded state or a not-yet-loaded state.

Making a Flex application work with an embedded SWF
To use the variables and functions in a SWF that’s loaded in your application with a SWFLoader, you need to make sure of these things:
  • import mx.managers.SystemManager
  • Make an instance of a SystemManager, i.e. something like public var loadedSM:SystemManager
  • Make sure the SystemManger instance gets binded to the properties of the embedded SWF with an intializer function that contains a line like loadedSM = SystemManager(SWFLoaderID.content)
  • Make sure that the <mx:SWFLoader> contains a complete property that will call the function that initializes the SystemManager

That being said, here’s a sample for a simple “application” (such a big word for such a silly thing), where clicking on the button changes the text in the Label component:

after clicking


Inner Application Code
<?xml version="1.0" encoding="utf-8"?>

 
  <![CDATA[
   public function pressed():void

   }

And here’s code for an outer application, which has a button that will change the text of the embedded SWF’s label:

Outer Application Code
<?xml version="1.0" encoding="utf-8"?>

 
  <![CDATA[
   import mx.managers.SystemManager;
   public var loadedSM:SystemManager;
   
   // initialize variables with information from the loaded application
   private function initNestedApplicationProperties():void
   {
    loadedSM = SystemManager(myLoader.content);
   }
   
   protected function useEmbeddedPressed():void
   {
    loadedSM.application["label"].text = "Hello, outer application!";
   }
  ]]>


So, to reference the functions or variables of an embedded SWF, you’ll want to preface variable/function names with loadedSM.application["variable/function name"].

Edit
Okay, so here’s something else I found somewhere while searching:

There is not support for using the Flex Framework in an AS only project. While it is theoretically possible, you would have to manually bootstrap a lot of the application initialization code that Flex handles (something which would be rather complex).

If you want to play with the framework, but don’t want to layout your app with mxml, then the easier way is to create a simple mxml file that contains an application tag, and a script tag that then points to your code.

This way, the application tag will take care of all of the initialization, and you can then leverage the framework via ActionScript.

SO I GUESS IT’S BETTER TO AVOID USING FLEX COMPONENTS IN AN ACTIONSCRIPT-ONLY PROJECT.

Which kind of begs the question, what’s the difference between Flex and Flash anyway? I mean, I guess I’ve kind of been running around in general because I still don’t quite get the difference between Flex and Flash and Flash Player and MXML and ActionScript and what, even Adobe Air.

So! Let’s have definitions:

Application/Application software: software designed to help a user perform task(s), i.e. writing papers, drawing stuff, listening to music.

Web application: an application that’s accessed over a network like The Internets. It can also refer to an application that’s coded in a browser-supported language, like something that’s a mix of JavaScript and HTML. These are good because web applications can be updated and maintained without distributing and installing software. Examples of web apps: e-mail, online retail sales, wikis.

Desktop application: I assume that an application is a desktop application unless it’s explicitly stated to be a web application? So like, a desktop application for listening to music would be iTunes, but a web application for listening to music would be…like…Pandora.

Software framework is an application skeleton where the application has already defined parts that are the most important – i.e. a software framework for a bank would already have withdraw/deposit methods coded up. They’re a special case of software libraries in that they’re re-usable and have other special features, i.e. inversion of control, which is when the framework dictates the flow of control, and not the caller. Frameworks can also be extended/added to, but not modified.

Rich Internet Application (RIA): “A web application that has many of the characteristics of desktop applications,” which are delivered by way of a browser or browser plug-in, of a sandbox, or a virtual machine. A user would have to install a software framework before using the RIA, and the framework would download, update, verify, and execute the RIA. The three most common platforms for RIAs are Adobe Flash, Java, and Microsoft Silverlight.

Runtime environment: software that supports the execution of computer programs. So, you can’t run a Java program if you don’t have a Java Runetime Enfironment (JRE).

Flash Player: a runtime environment that runs Flash-platform content and applications. Flash Player is used for viewing animations and movies using a web browser. It runs SWF files, which can be embedded in websites. It was originally meant to display 2D vector animations (the likes of which you can build in Adobe Flash), but since then has been also used for making RIAs, and streaming video and audio. And games, and GUI.

Adobe Integrated Runtime (AIR): a runtime environment for building RIAs with Adobe Flash, Adobe Flex, HTML, or Ajax, which can be deployed as desktop applications. So, for a user to use an application executed in Adobe AIR, Adobe AIR must be installed onto the user’s computer at operating-system level. (On a side-note, I noticed Adobe AIR being installed on my computer while installing Rosetta Stone…I guess that’s why.)

Flash: Refers to Adobe Flash (the program), and the Flash Player. Adobe Flash makes flash files, which Flash Player runs.

Adobe Flex: an SDK released so as to develop/deploy RIAs based on Adobe Flash. Flex applications can be written using Adobe Flash Builder (which you have to give moneys for), or just by using a text editor/Eclipse and a Flex compiler. Flex applications are made using MXML and ActionScript. It’s meant for creating RIAs, and provides a lot of useful things, like pretty-looking UI components (buttons, color pickers, grids). Adobe Flex also outputs an SWF file.

ActionScript (3.0): an object-oriented language used for developing websites and software. It outputs into an SWF file, which is then embedded into a website and run by Adobe Flash Player. It was initially meant to control the 2D vector animations made in Adobe Flash.

MXML: used in combination with ActionScript to create RIAs. Flex uses MXML for the layout of the application, and ActionScript for the programming logic. MXML is eventually compiled into ActionScript. Adobe Flash Builder can compile MXML into a SWF file.

SWF file: the output of compiling ActionScript. SWF files can be used by Adobe Flash Player (for the web) and Adobe AIR (for the desktop).


I think the reason for all of my confusion was that “Flash” is apparently used for both Adobe Flash Player, the runtime, as well as Adobe Flash, the authoring environment, as well as like, “Flash games,” which kind of makes you think of some game that’s made “in Flash,” when really it’s written in ActionScript, and compiled to an SWF that is run by Flash Player. And when people compare Flash and Flex on the internets, what they’re really comparing is Adobe Flash and Adobe Flash Builder, probably, not a set of two different languages.

Dec 28th, 2010

Though both Kelly and Nandrew made faces when I said I was trying to learn this

In General:
Flex applications are comprised of MXML and ActionScript. MXML/Flex is used for the application interface, and ActionScript is used for the programming logic. Compiling a project yields a SWF file, which is referenced in an HTML page. When someone looks at the page, the SWF file is downloaded and rendered by Flash Player.

When you create a Flex Project in Flash Builder, you’ll automatically get an MXML file with the same name as the new project.

This is the initial code you get in that MXML file:

<?xml version="1.0" encoding="utf-8"?>

 
      (non-visual elements like services, value objects are placed here)


<s:Application> defines the container for the application; it’s the root tag in the XML document.


MXML and UI
MXML defines what components are in the UI – Buttons, CheckBoxes, DataGrids (which show lists), etc. When you look at the MXML file in Flash Builder, two tabs will appear, which will allow you to alter the layout of the application. You can use the Design tab (when you look at your .mxml file) to add buttons, etc. from the Components bar in the bottom-left, or you can write it buttons manually into the Source with tags that look like <s:Button>, <mx:DataGrid>, etc…

buttons and stuff, yeeeeh

Buttons and whatever you make in the Design tab get converted into MXML. So, looking at the Source tab, this code was generated from my lolsy layout above:

<?xml version="1.0" encoding="utf-8"?>


It doesn’t matter what order the components are in within the MXML file because their positions are determined by the x and y attributes. Other styles can be set this way, like colors – you can view all of them by clicking inside a tag and pressing CTRL+space. Alternatively, all the styles can be done by CSS.

You can get even more information about the components by clicking Help > Dynamic Help and clicking on whatever component you want.

MX and Spark Components
If you take a closer look at the tags for the components added, you’ll notice that their namespaces are different – for instance, the Datagrid tag <mx:DataGrid> has a namespace of mx, whereas the button tag <s:Button> has a namespace of s.

This is due to the fact that Flex 4 introduced a new set of components called Spark components, and uses them along with the older components, called MX Components. Some advantages of the Spark components are that they’re easy to custom skin. In general, Flex 4 lets you choose which components you want to use, and you can mix and match, though use Spark components when you can.

I found more about Spark components and general Flex 4 stuff here.


Making the Button Do Something
To make the button do something, you’ll need to take advantage of the fact that buttons emit “click events” whenever someone presses them. Change the <s:Button tag to include a property click="button1_clickHandler(event)". Flash Builder should fill in that function name as a default, and also add brand new code right beneath the <s:Application> root node, which looks like this:


 <![CDATA[
  protected function button1_clickHandler(event:MouseEvent):void
  {
   // TODO Auto-generated method stub
  }
 ]]>


You can make a pop-up show when someone clicks the button by replacing the commented-out code with Alert.show("LLAMAS"). Flash Builder should automatically import mx.controls.Alert before the function definition, so that the code should now look something like 


 <![CDATA[
  import mx.controls.Alert;
  protected function button1_clickHandler(event:MouseEvent):void
  {
   Alert.show("LLAMAS");
  }
 ]]>



Then you can (build and) run and click the button to your heart’s content.

Creating Application States
In a website, you make different files for the different pages that you visit, but an application instead has states. You can create them by going back to that Design tab, then looking for the new States window and right-clicking to create a State.

Switching Between Application States
You can switch between application states by setting currentState to something, like currentState = state2. Placing this in the code for the click event of the button like this:

protected function button1_clickHandler(event:MouseEvent):void
 {
  Alert.show("LLAMAS");
  currentState = "State2";
 }

Will make it so that clicking the button labeled “button!!” will bring you to the page with the llama Image. Though there’s no way to get back to the first state, for now…

In conclusion, here is all the code for this:

<?xml version="1.0" encoding="utf-8"?>


 
  <![CDATA[
   import mx.controls.Alert;
   protected function button1_clickHandler(event:MouseEvent):void
   {
    Alert.show("LLAMAS");
    currentState = "State2";
   }
  ]]>
