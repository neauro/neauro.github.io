Notes From Messing Around With Processing Tutorials

Jun 8th, 2011

my first great processing creation

Processing is apparently supposed to allow artists to interface with Java as if it were more of a scripting language, in case Java itself is too scary or complicated. (It’s also meant to allow you to do simple projects without the overhead of all the Java code and learning Java, and is a good gateway programming language.) Actually, thinking about it now, my first project with Java involving drawing a house, lifting pens and putting them down and changing colors and all. It was the most frustrating thing ever and I’m glad I stuck with CS anyway.

Little bundles of code for Processing are referred to as “sketches.” Here is the code in case you want to create a sketch to make tons of circles for yourself. (It is also Tutorial 1 on the Processing website.)

void setup() {
  size(480, 120);
  smooth();
}

void draw() {
  if (mousePressed) {
    fill(0);
  } else {
    fill(255);
  }
  ellipse(mouseX, mouseY, 80, 80);
}

It seems that setup() is called once, when the program starts running; draw() is called constantly, much like an update() function in game programming. You can also name functions with headers like void mousePressed(), which is a function which would be called every time the mouse is pressed.

After you’re done coding something with Processing, you can Export it into an applet, or an application for Windows, OSX, or Linux. You can also just easily save frames by calling saveFrame("name-of-output-picture####.png"), where the hash marks will be replaced with a numbered sequence.

Here’s sample code that deals with creating a Car object in Processing:

Car myCar1;
Car myCar2; // Two objects!

void setup() {
  size(200,200);
  // Parameters go inside the parentheses when the object is constructed.
  myCar1 = new Car(color(255,0,0),0,100,2); 
  myCar2 = new Car(color(0,0,255),0,10,1);
}

void draw() {
  background(255);
  myCar1.drive();
  myCar1.display();
  myCar2.drive();
  myCar2.display();
}

// Even though there are multiple objects, we still only need one class. 
// No matter how many cookies we make, only one cookie cutter is needed.
class Car { 
  color c;
  float xpos;
  float ypos;
  float xspeed;

  // The Constructor is defined with arguments.
  Car(color tempC, float tempXpos, float tempYpos, float tempXspeed) { 
    c = tempC;
    xpos = tempXpos;
    ypos = tempYpos;
    xspeed = tempXspeed;
  }

  void display() {
    stroke(0);
    fill(c);
    rectMode(CENTER);
    rect(xpos,ypos,20,10);
  }

  void drive() {
    xpos = xpos + xspeed;
    if (xpos > width) {
      xpos = 0;
    }
  }
}

And here’s an example of grabbing samples of color from an image:
for (int x = 0; x < img.width; x++) {
  for (int y = 0; y < img.height; y++ ) {
    // Calculate the 1D pixel location
    int loc = x + y*img.width;
    // Get the R,G,B values from image
    float r = red   (img.pixels[loc]);
    float g = green (img.pixels[loc]);
    float b = blue  (img.pixels[loc]);
    // Change brightness according to the mouse here
    float adjustBrightness = ((float) mouseX / width) * 8.0;
    r *= adjustBrightness;
    g *= adjustBrightness;
    b *= adjustBrightness;
    // Constrain RGB to between 0-255
    r = constrain(r,0,255);
    g = constrain(g,0,255);
    b = constrain(b,0,255);
    // Make a new color and set pixel in the window
    color c = color(r,g,b);
    pixels[loc] = c;
  }
}
Other miscellaneous notes:
  • width and height are variables which refer to the size of the sketch
  • size(x, y, option) will create a sketch of width x and height y, with an optional rendering mode. The default is JAVA2D, which does high-quality 2D graphics at the expense of speed. P2D refers to Processing 2D, which has simple graphics and fast pixel operations, so it’s good if you need to do thousands of shapes. P3D (Processing 3D renderer) can produce 3D graphics even without the use of a library, and is meant for speed and pixel operations. OPENGL mixes use of Sun’s Java for OpenGL library for faster rendering, and Processing’s APIs for application export (for which the P3D renderer is a simpler solution, if you don’t want your users to see the whole “Are you sure you want to trust Sun Microsystems” dialog). Finally, size(x, y, PDF, "pdf-name.pdf") will draw to a file rather than a screen.
  • You can load images easily in Processing, but it takes a while, so it’s best to do it only once; i.e. not in draw() but in setup().

HCI Project: Kibisis

Jun 8th, 2011

logo by joy kim!

For Introduction to HCI, we had a quarter-long project to design a ubiquitous technology. I think that everyone ended up doing mobile phone applications, including us. We designed Kibisis, which is a mobile payment application. Its main themes are:

  • Connect your credit cards, debit cards, checking and savings accounts, etc. to one application, so that you can consolidate all your methods of payment
  • Keep track of your loans and debts to your friends, useful especially for college students who may split rent or loan money to their friends for dinner
  • Transfer your money directly, remotely, and securely from your Kibisis account to the Kibisis account of other people, removing the need to stop by banks to cash checks, or to get exact amounts of cash and then meet up to hand it off

Listed above (and here, again) is the website for our project, which contains all the reports we wrote, which details our testing and design process. I made the website – it was a lot of fun. Game: Get Across

Jun 8th, 2011


Yesterday was our last presentation for Games Capstone. All in all, it was a pretty good quarter – I’ve learned a lot about clientside and serverside game programming using Flixel and Player.IO, as well as how to solve (or even better, avoid dealing with) some of the most annoying problems.

You can check Get Across on Facebook, and see the other social games on this webpage.

First,

An Overview

Get Across is a multiplayer game with tile-based movement. The objective is to get across (whoa!) a sort of terrain, where different parts of the terrain take more energy to get across, i.e. getting across a mountain takes more energy than getting across a hill, which in turn takes more energy than getting across normal flat ground. The player can be one of several classes, and each class has its own special abilities: Crafters can get across water by building bridges, Planters can reduce energy usage, Cooks can replenish AP or lure monsters away with Monster Bacon, etc.

terrain of the first tutorial

The idea is that you and your friends could work together to solve boards, and be scored on things such as how long it took you to solve the board together, how much energy and other resources you used, etc. There could be leaderboards for different teams, and you could also make boards on your own and have your friends “conquer” them. Each terrain would be kind of like a puzzle that you would be able to solve in different ways, given your different abilities, and different “cards” that you could take into the level, which would have special effects like making mountains explode, or allow players to switch positions with each other, etc. Like a board game.


Implementation

We – Ji, Charlie, and myself – used Flixel, as well as Player.IO for the multiplayer stuff and saving players and states to the database.


Problems

One of them was our thread abortion problem, which arose out of the blue at some point and refused to leave.

i hate you more than i hate shrimp and scallops

Eventually, we found this thread on the Player.IO forums which was made pretty much the day that we started having this problem unbearably, like 90% of the time we started running the game. I am 90% sure that all of our thread abortions were the cause of the error discussed by Oliver:

That error precisely means that one thread was caught using too much cpu, and that it was aborted as a consequence. We have to ensure that all events are handled swiftly (<100ms), otherwise the server will perform really badly. I’ve specified earlier how our new measuring system works: first it uses global time, and then it switches over to thread-specific time, so the other load on the machine *shouldn’t* make a difference… unless of course it’s very high load, in which case all bets are off.”

Every player was saved in a “Quest” object, which contained that player’s position on the board, their AP, the amount of resources they had, as well as the state of the board, tiles and monsters and so forth. Some of these were stored in DatabaseArrays, and some of them were DatabaseObjects in DatabaseArrays…

so much embedded

One unforeseen problem with the arrays was this weird circular references error, which happens if you try to create a new object, and then set an array in that object to be the same as another object’s array. Due to some weirdness, in order to copy an array over, right now the accepted workaround seems to be to just copy that array element by element… :/ Which could have contributed to our 100+ ms. To change this array structure completely would have taken more time than we had, especially since we had already implemented most of the game, but I imagine we could have maybe substituted some of the objects for strings. For instance, every monster on the board was a DatabaseObject containing an x-position, a y-position, a “type” (either weak or strong), and a health variable; instead this could have just been a coded String.

Anyway – leaving Get Across now, I’m left with a lot of Lessons Learned.


Paper Prototyping and Testing is Essential

The first month or so of our Capstone class was just paper prototyping and testing, which I feel was met with a lot of skepticism on our part. Paper prototyping for Get Across was pretty easy (if tedious): we just had a grid, and a couple other paper bits to represent mountains, rivers, hills, etc., which we rearranged as necessary.

paper protyping get across

Paper prototyping was also a theme of the HCI class that I took this quarter, for obvious reasons: the more you start building something, the harder it is to change. Paper prototyping helped us learn early which mechanics would work and which wouldn’t. For example, one “social” aspect of the game would make it so that, of two players playing a map, one player would have to sit on a button and wait for the other player to go through the gate the button opened; but we learned early on that, especially in an asynchronous game, this is an…inadequate idea. When someone logs onto a game, they want to be able to do something; if they’re just sitting and waiting for someone to move, unable to force them to do so, then it’s regarded as a waste of time. We eventually made it so that buttons only had to be pressed once for a gate to open permanently, which isn’t novel, but what we were limited to in time this quarter.

Paper prototyping was hard to use for some aspects of our game, however – for instance, we were thinking that we could use a CityVille-like usage of energy which would come back in time, and in play-testing sessions, since we replenished AP instantly when people lost it, people just didn’t feel like their AP was rare enough to save. In retrospect, we could have just made it so that people would wait like 10 seconds or something when they ran out of AP.


Testing and More Testing and More Testing is Also Essential

There were some days when we came to class having a shambled, three-legged version of a game, and it was on those days especially that I didn’t want to let anyone see the game; but even if all of our functionality wasn’t implemented yet, we were still able to discover a lot of interesting things that we just wouldn’t have had the insight to realize on our own. Here is a sample of the (very long) list of things we gathered during one playtest last month:
  • people will start moving with the mouse before the arrow keys
  • people don’t notice AP diminishing
  • “toggle on and off the bridge is a little confusing” – when bridge is selected, instead of putting the yellow square, put an overlay of the bridge? also, turn off the bridge ability once a bridge is made
  • frustrating to wait for AP to come back when you’ve lost it
  • how many clicks do people use to go through the menu, and as they play? try and minimize number of clicks
  • and so on


Having a Schedule Would Have Probably Made Our Lives Easier

The way that we worked on Get Across was a little haphazardly – we pushed out the features that we wanted to, versus the ones we probably should have worked on, which made it so that our Facebook integration was close to nothing. Which brings me to the next point:


If It’s Supposed to be a Facebook Game, Build It on Facebook From the Very Beginning!

It seems obvious, but this particular Capstone advised this course very strongly: release on Newgrounds, then on Kongregate (which has a larger populace), then on Facebook. In the end, we only did Kongregate and Facebook (since Player.IO supports these two, and not Newgrounds), but not focusing on Facebook integration from the very beginning (allowing you to see your friends, or to send invitations to them immediately) made Get Across’s Facebook release very weak.



And those are the writing-worthy ones that I can think of. Simple, yet effective lessons. In the end, working with Player.IO was fun and it definitely served its purpose, and the only thing keeping it from working perfectly for us was the learning curve.

Knowing slightly more than I did before this game, and tons more than I know than when I worked with Jimmy and Leilani to put together Bonding Time, I am really excited to start my next game project. :D Just gotta think about what it will be.

Bridesmaids, and Stories With a Razor-sharp Focus and Relevant Characters

Jun 8th, 2011

(source)

In (somewhat premature) celebration of making it through the Computer Science department and retaining our artsyness, yesterday Jimmy and Mari and I went to Wann Izakaya for food and also saw Bridesmaids, which Jimmy wanted to see because the director apparently studied the art of story under another story guy that Jimmy really admires.

Something like that…

[spoilers ahead]

The movie was pretty funny. There were a ton of parts that were so awkward it was painful to watch (in a good way, I suppose). I didn’t mind to much while watching it, but afterward, talking about the story with Jimmy and Mari on the way home, it seemed to us that there was still a lot to be desired.

Jimmy for one is a huge proponent of focus in stories, and thinks that every movie should have an armature, which is essentially “the moral of the story.” The armature for The Godfather, for example, is that “Your family defines you.”

Throughout the entirety of Bridesmaids, the main character, Annie, appears to struggle with her negative outlook and lack self-confidence. She’s in a relationship with a clearly douche-y guy who doesn’t care about her, she struggles with another character who appears to be stealing her best friend away, she ends up pushing away the people who do care about her because of her own pessimism. Things begin to look up for her when she does decide to think positively, which occurs after her last, enormous mad outburst, and (apparently) due to one of her other “friends” tackling Annie on the couch and slapping her until Annie fought for herself.

If you’ve been depressed and thinking negatively for months, Jimmy pointed out, thinking positively doesn’t come easily. It takes a little more than a fat woman to beat you up on your furniture to make you think positively.

And once she does think positively, the solutions to all her problems just fell into her lap. Sure, she attempted and seemed to fail at apologizing to her love interest after wronging him – but that’s only a small step out of the comfort zone that she’s been hissing herself into the entire movie. It’s true that she never had to overcome anything greater than herself for the entire movie, but if doing so was so easy, why didn’t it just happen fifteen minutes in?

If a good movie should have focus, if every scene of it should mean something (much like how every sentence should mean something in a short story), and a big theme, then this movie fell something short of “good.” There seemed to be a lot of things that didn’t really mean anything – for instance, why was Annie a baker? The majority of the things that she did in that movie were completely unrelated to baking, and she only ever baked anything in two scenes. The first time that she baked something, it was after she met her love interest, and because he admitted he was a neighbor, I thought: Would she give it to her love interest? But no: she just looked at it, then ate it.

What is that supposed to mean? Does she bake to try and calm herself? If so, why doesn’t she do it the other times that she panics during the movie? Was she thinking about giving it to the love interest, and then was too afraid to do so?

Why was it important for Annie to be a baker? Was it just so that she could have a failed start-up, to show that she had ambition but had been broken by her failure? If so, couldn’t there have been a better way to do this, without giving her a completely superfluous label as “baker”? Or, if she definitely is a baker, why wouldn’t this show through in other aspects of her life? She doesn’t have recipe books. She doesn’t comment about the food that she eats. She doesn’t appear to have anything super baker-y in her room, which in a few shots is nice and elegant but completely lacking in description of the type of character that she is. People’s rooms can be the greatest indicator of what type of person they are – clean, messy, cluttered – my roommate’s typography decorations and shelves of comic books speak differently than my wall of folded-up paper chopstick-covers and small shelve stuffed with PS2 games. Mariko has a wire shelf of animu and Sakuracon badges. Jimmy has a triplet of monitors for his computer, Molly’s desk and windowsill is lined with classics and books about healthy eating.

As Jimmy texted, a couple minutes ago: Never have a batman movie that can also be a superman movie. Character must be intrinsic to plot. Annie could’ve been anything.

Which means, really, that she wasn’t anything.

There were a lot of other unnecessary parts – Annie’s mother maybe finding a new partner at the end, two of the bridesmaids becoming completely irrelevant at the end of the movie, so on. Anyway, it’s a funny movie.

Her Will (a Short Story for AU III: Legacy)

Jun 6th, 2011

AU!  woooo!

Last Wednesday was the third release party for AU, and also the last for this year, and also the last for its founding editors, who will all be graduating.

The party was small this time around, only about seven or so people, despite the advertisement of funfetti cookies and other handmade deliciousness. Apparently a ton of people submitted short stories for this quarter’s “Legacy” theme, and most of those stories were right on the border of 3,000 words. As AU’s size is apparently limited by how many pages the editors can physically staple together, they could only accept 6 submissions.

cover by katherine tung, once more

My submission is a short story titled Her Will, and you can find the full thing here.

When I heard about the theme, I searched Wikipedia and came to the conclusion that something playing off an inheritance or will would be a good idea. I also thought that it might be neat if a “will” were more literal – that is, that someone who died in some alternate universe would be able to possess some person to carry out a last task. Eventually this idea was trashed, but, meh. I may come back to it again.

I think that it’s fair to say I’ve never written short stories before AU; I just never really had the idea for it, nor the urge. I always found it hard to squeeze all my ideas into one short story. But I’ve discovered that writing short stories are surprisingly, uniquely fun – there’s something about it that’s almost like design. There has to be a meaning to every word, every sentence. For this story, I went beginning to end multiple times trying to cut out everything I couldn’t immediately justify, and strengthening everything else that was left. Though I would call the story far from perfect (when does any story ever reach that point?), even now I can’t think of one sentence that I would feel comfortable removing. I feared that some of the sentences would start to sound blunt and lifeless, but hopefully that’s not the case in the final draft.

I’m going to miss AU! Hopefully it will be taken up by new people next year, and I’ll be able to contribute since I’ll still be a UW (Master’s) student.

Get Across: Kongregate Release

May 23rd, 2011

the first 200-some players of our silly capstone game

It’s been about 24 hours since our first online release of Get Across. Our capstone class recommends moving from Newgrounds, to Kongregate, but then to Facebook, but it’s taken us such a long time to learn Player.IO that we skipped straight to Kongregate, as Player.IO already supports this..

Still, our release was (to me) a little rushed, and I would have loved nothing better than to clutch the game even tighter and keep anyone from seeing it at all. We have presentations on Tuesday on what our first round of releasing reveals to us about the game, and I haven’t parsed through the data that we’ve collected yet, but an onslaught of a mysterious error regarding null quest objects made me really self-conscious about what was going on. I don’t want anyone to play a game that is still buggy. Nononononono!

It’s hard to debug from Player.IO’s error log, though it is very informative concerning what errors popped up, whether they were serverside or clientside, when they happened, and how often.

just look at those graphs, dude!

Some of the players on Kongregate have been really helpful in leaving comments about their thoughts and the bugs they’ve encountered; but there are about 6 of these comments, and 200+ registered players, most of whom didn’t really make it through the tutorial levels. I have a good guess why: the tutorials take forever to load.

I’m not sure why, but I’m guessing it might be because we have a lot of objects in the database, and a lot of the properties of those objects are nested objects or arrays, which must take longer to retrieve. Seeing as tutorial levels are pretty essential for hooking in initial players, I’m hoping to make it so that the tutorial levels are completely static and client-side – no saving data into the database, nothing. If a player comes back, they’ll just have to start over. Hopefully this will make it so that we only really save “real” player data in the database as well, and keep out the clutter.

So even though I really wouldn’t have wanted to release the game this soon, this little bit of playtesting illuminated something that we otherwise would have learned at the last awful minute, for the final Facebook release. Which is nice.
