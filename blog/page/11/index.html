Paula Deen’s Easy Cheesecake Cupcakes

Aug 20th, 2012


The other day HC came over and we decided to make cupcakes. Given my limited store of ingredients, we went with cheesecake, that being the type of cupcake that requires little investment in terms of icing and ingredients that I didn’t posses.

The recipe is sourced from what I now realize is Kraft’s Recipes. We changed the butter extract to vanilla extract. The recipe is for 12 cupcakes, but we ended up making around 18. Also, we ended up not putting any jam or anything on the cupcakes, but they were quite sweet on their own.

Ingredients
  1. 2 pkg. (8 oz. each) cream cheese, softened
  2. 1 cup granulated sugar
  3. 1 tsp. vanilla extract
  4. 2  eggs
  5. 12 vanilla wafers
  6. 1 cup seedless raspberry jam
  7. 1pt. fresh raspberries
  8. 2 tbsp. powdered sugar for dusting

Instructions
  1. Place a paper cupcake liner in each of 18 muffin cups.
  2. Beat cream cheese until fluffy. Add granulated sugar and butter extract, beating well. Add eggs, one at a time, beating well after each addition.
  3. Place a vanilla wafer, flat-side down, in each muffin cup. Spoon cream cheese mixture over wafers, until paper lining is about 3/4ths filled. Bake for 20 minutes.
  4. Allow tarts to cool completely. When cool, top each cheesecake cupcake with 1/2 Tbsp. of raspberry jam and fresh raspberries. Dust with powdered sugar.

Since the vanilla wafers we bought were a little too small for the paper liner, we experimented with making half the liners filled with whole wafers, and half with crushed. (By the way – vanilla wafers are easily crushed by placing them inside a glass mixing bowl, and mashing them with the flat bottom of another glass.)


In the end, the crushed wafers ended up quite crumbly, so whole wafers really are probably the way to go.

How do JavaScript functions deal with arguments?

In JavaScript, every function has an argument list of variable length, so a function that takes and prints 1 argument can still be passed 3 arguments, and will complainlessly print out the first argument.

A function can look at all the arguments its been passed by looking at an arguments object, which is like an array (with indexes, and a length, though you can’t push/pop into it, or split it).

Source: Function.apply and Function.call in JavaScript



What’s the difference between JavaScript call and apply functions?

What call and apply do is change what the keyword this references at run-time.
x = 10;

 function function_name() {
   alert(this.x);
  }

 function();

In the previous, this references the global namespace, and the output is 10.

x = 10;
 object = { x: 5 };

 function function_name() {
   alert(x);
  }

 function();
 function.call(object);

This time, the first function call outputs 10, whereas the second function call outputs 15. this in the second function call uses object as the namespace, and refers to object’s value of s.

You can also pass arguments to a function using call, as happens in the following.

x = 10;
 object = { x: 5 };

 function function_name(message) {
   alert(message);
   alert(x);
  }

 function("invoking f");
 function.call(object, "invoking f with a call");

Finally, what’s the difference between call and apply? They both change the namespace at run-time, but apply’s second parameter always needs to be an array.

x = 10;
 object = { x: 5 };

 function function_name(message) {
   alert(message);
   alert(x);
  }

 function("invoking f");
 function.apply(object, ["invoking f with an apply"]);

Apply is the more interesting of the two; because it can use an array, it can work with JavaScript’s arguments object.

What is currying?

To curry is to take a function which accepts multiple arguments, and to convert it into one funtion that takes one argument and returns another function. The last thing returned should be the result.

What are the advantages of currying? If functions are in a curried form, they may be easier to apply partially. Also, the parameters that have been “left out” when a function is curry-fied may not even be known at compile time, but will be at run time.

In general it seems to be that currying in general is done out of convenience rather than necessity, to create simpler-looking methods.

JavaScript uses closures to save the partial state

How do closures work in JavaScript?

A closure describes a function, and the environment that contains the variables that the function can reference. Because the closure is a stack frame that is not deallocated when a function returns, it also contains the local variables that remain “alive” after a function has returned. A closure is used every time a function is declared within another function.

function say_hello(name) {
    var hello = "hello";
    var hello_alert = function () { alert(hello + name); }; // closure created!
    return hello_alert;
  }
  
  hello_arupaka = say_hello(arupaka);
  hello_arupaka(); // alerts "hello arupaka";

In most languages, after a function (like say_hello(arupaka)) returns, the local variables are destroyed; but, when calling hello_arupaka(), the locally-created text ”arupaka” is still available. Closures!

function setup_globals() {
    var num = "3";

    var alert_num = function () { alert(num); };
    var increment_num = function () { num++;  };
    var decrement_num = function () { num-- };
    return hello_alert;
  }

So what are the closures of alert_num, increment_num, and decrement_num above, and how do they relate? Because closures are set up every time a function is called within a function, they all have shared access to the same closure.

Note: if you were to call setup_globals one more time, all the inside functions would be overwritten.

Next closure example: what does testList() output?
function buildList(list) {
  var result = [];
  for (var i = 0; i < list.length; i++) {
    var item = 'item' + list[i];
    result.push( function() {alert(item + ' ' + list[i])} );
  }
  return result;
}

function testList() {
  var fnlist = buildList([1,2,3]);
  for (var j = 0; j < fnlist.length; j++) {
    fnlist[j]();
  }
}

Answer: 3 alerts, all of which read item3 undefined. This is because all functions created by result.push( function() {alert(item + ' ' + list[i])} ); have shared access to the same closure. The array containing these functions in turn looks like this:


And when testList calls fnlist[j](), each list[i] uses the same value of i (which was incremented to 3) and list[3] is undefined. Each cell alerts item3 because that was the last value assigned to var item.

What’s the difference between event bubbling and event capture?

First: DOM (Document Object Model) is a convention for representing and interacting with objects in HTML, XHTML, and XML. When a page is rendered by a browser, that browser parses the HTML from the web server into an in-memory DOM. Different web browsers have their own internal model of DOM. The DOM views an HTML document as a tree.

DOM events allow programming languages like JavaScript attach event handlers to tree nodes (different elements on a page), and react when users click/mouseover those nodes.

So, what happens when you want to attach an event handler to a couple different elements on a webpage? Rather than define separate handlers for each element, you can have the events work with a node that contains those elements.

In order to find the containing node, events need to be able to “propogate,” or move through the DOM node tree. In event bubbling, the event is first handled at the inner most elements, and then propagated to the outer elements. (A great example of this is here!) Event capturing is the opposite: the event is first handled by the outermost element, and then propagated to the inner-most element.

Only event bubbling is supported by all major browsers.

Sources: Wikipedia: Document Object Model, W3Schools: HTML DOM Node Tree, Event Propagation, StackOverflow: What is event bubbling and capturing?

Wicked Plants, Amy Stewart

Aug 16th, 2012

cover of wicked plants (source)

A tree that sheds poison daggers; a glistening red seed that stops the heart; a shrub that causes paralysis; a vine that strangles; and a leaf that triggered a war. In Wicked Plants, Stewart takes on over two hundred of Mother Nature’s most appalling creations. It’s an A to Z of plants that kill, maim, intoxicate, and otherwise offend. You’ll learn which plants to avoid (like exploding shrubs), which plants make themselves exceedingly unwelcome (like the vine that ate the South), and which ones have been killing for centuries (like the weed that killed Abraham Lincoln’s mother).

Menacing botanical illustrations and splendidly ghastly drawings create a fascinating portrait of the evildoers that may be lurking in your own backyard. Drawing on history, medicine, science, and legend, this compendium of bloodcurdling botany will entertain, alarm, and enlighten even the most intrepid gardeners and nature lovers.

I generally adore plants and like hoarding information about them, so this book was super enjoyable for me. The illustrations are beautiful, even the disturbing ones that show fungus growing out of people’s nostrils or hands holding bouquets of eyeballs. It’s easy to read and the muted textures and colors on the paper made it especially fun somehow – though I love my Kindle, this particular book wouldn’t have been the same in .mobi format. It makes me wonder whether mass paperback of the future will continue to make their niche in an increasingly electronic world by experimenting more in form and color, offering what websites just can’t yet.

Some interesting finds:

  • Killer Algae: a plant mutated potentially by aquarium UV lights, whose entire body (fronds, stems, and all) comprise one cell. Even so, it apparently covers twelve thousand acres of the ocean floor. What.
  • Cogon Grass: a grass with “lures fire” with its high flammability and causes it to burn hot on neighboring competition. Cogon seeds then grow afresh and strong from the ashes.
  • Sandbox Tree: “The sap is so caustic that it can kill fish or be used as an arrow poison. But you have the most to fear from the fruits, which explode with a loud bang when they are ripe. Its poisonous seeds can fly up to three hundred feet, earning it the nickname ‘dynamite tree.’” What.
  • Celery:  apparently, to combat a common fungal predator, celery can produce phototoxic compounds at its root and “farmworkers and handlers of celery routinely get burns on their skin that show up under sunlight…” What.

Needless to say, now I feel like every little green thing on the street is just waiting to inject me with deadly neurotoxin. (Thanks, availability heuristic?)

Some HTML5/JavaScript Links

Aug 16th, 2012

Having done few HTML5 things, I’ve dug up some tutorials and interesting links.


HTML5 canvas - the basics

I guess the first cool thing about HTML5 is this canvas thing, which is basically like a drawing pad that HTML5 bestows upon you, eyes glowing with love. “Go,” it says. “Use this to create art with your present armory of JavaScript.”

A canvas object has a bunch of contexts, the same way a drawing pad has a lot of pages. JavaScript can be used to draw onto a context. Most browsers allow JavaScript to draw 2D things; some have experimental 3D contexts.

Unlike a drawing pad, once you’ve decided to use a canvas’s context, that’s the only context you can use for that canvas, ever. It’s kind of like you tore out a page from the drawing pad and all the other ones withered away in anguish.

Interesting things canvases can do in HTML5:
  • draw lines, ex. canvas.lineTo(xPos, yPos)
  • draw rectangles, ex. canvas.fillRect(xPos,yPos,width,height)
  • draw images, ex. canvas.drawImage(imgSource, xPos,yPos)
  • apply pixel manipulations to those images, ex. invert all colors
  • draw text (only in recent WebKit builds)
  • draw dropshadows and gradients for all your Web 2.0 needs


Understanding save() and restore() for the Canvas Context

When searching around for examples of canvas usage, I saw references to canvas.save() and canvas.restore() which were nicely explained in this article.

In brief, every canvas contains a stack of “drawing states.” Drawing states are comprised of a lot of data, including the current transformation region, the current width of line being drawn, the current front, and so on.

If you want to draw a shape like a diamond (or rotated square), the steps to do this involve:
  1. Make a canvas, retrieve its 2D context
  2. 2Dcontext.save()
  3. Rotate the context
  4. Draw a square on the context (which, since the canvas is rotated, will be diamond-shaped)
If you then wanted to draw a new shape, like a regular square, you could easily pop the present drawing state with 2Dcontext.restore() to get back to an un-rotated coordinate system. The diamond should stay the way that it is presently.

Similarly, objects on a canvas can’t be “just moved;” if you’re translating an object across the canvas, the steps go
  1. Clear the canvas
  2. Draw the object in its new position
  3. Repeat

JavaScript function declarations vs. function expressions

In the previous tutorial, I encountered my first instance of a function expression. Essentially, a function definition looks like
 function function_name() {
   // stuff
  }

whereas a function expression looks like
 var = function function_name() {
   // stuff
  };

Some links I found which discussing declarations versus expressions are Function Declarations vs. Function Expressions and Named function expressions demystified, and the differences include the following:

  • Function expressions can be named or anonymous.
  • Function declarations are parsed and evaluated before any code is executed, whereas function expressions are evaluated at run-time. Therefore, if you call a declared function at any time, it should be okay, whereas function expressions must be defined before being called.
  • Function declarations should not be used in if-statements. In contrast, function expressions are okay to call in if-statements, and are sometimes used this way to allow for best performance (“don’t define this function unless it’s needed”).
  • Function expressions mimic what JavaScript is actually doing, which is creating an object every time a function is defined.

In general it seems that function expressions are the favored creature.

Objectified

Aug 13th, 2012


Objectified is a feature-length documentary film examining the role of everyday non-living objects, and the people who design them, in our daily lives. The film is directed by Gary Hustwit. Objectified premiered at the South By Southwest Festival on March 14, 2009. (via Wikipeda)

Objectified has been languishing on my Netflix Instant Queue for the good part of a year or so. I finally watched it, and was glad! The film brings together a variety of designers from Across the World, who all talk about what they personally feel is Good Design. It’s refreshing to watch designers talk so passionately about chairs and personal marks on the world and shaving away at the superfluous “fat” of a toothbrush design until they eliminate the concept of a toothbrush from oral health completely. It’s amusing to see one designer swear that he’s always “lived in the future,” while another reaches into the past for a modern cellphone design inspired by childhood memories of peeling potatoes.

In the end, I was relieved when the movie moved from “NEW THINGS! Aren’t they cool? Buy all these new cool things!” to “Do we really need new cool things? I mean, just look at all the things that we already have.”

just…look at them (source)

The film points out the great problem of sustainability in product design, how it’s not just about recycling/upcycling but how sustainability must guide design from before the beginning to beyond the end. Why are you designing this thing? What will it add to the world that doesn’t already exist? And what will happen to the thing, when someone designs a new thing five seconds later?

Visual Merchandising, Tony Morgan

Aug 8th, 2012

cover of visual merchandising (source)
A great introduction for retail students, this book offers a user-friendly reference guide to all aspects of visual merchandising and covers both window dressing and in-store areas. Using examples from a range of stores from fashion emporia to supermarkets, the book offers practical advice on the subject, supported by hints and tips from established visual merchandisers. It reveals the secrets of their tool kit, and information on the use of mannequins, the latest technology, how to construct and source props, and explains the psychology behind shopping and buyer behavior.

Presented through color photographs, diagrams of floor layouts, and store case studies, and including invaluable information such as a glossary of terms used in the industry, Visual Merchandising is an essential handbook for anyone working in and learning about this exciting area.

This came up as a hit when I searched the public library database for books about stores and merchandising, which I was in turn searching for as a resource for free-to-play game design. As NPR uncovered, free-to-play games are a lot like stores; you see the nice displays outside, shuffle in and around for a while, and if you discover something that you like, something that will, perhaps, make you feel like a better human being, you pay money to obtain that thing.

Electric Rest details some other possible applications of store design to free-to-play games, and I thought I would try and find some myself. Visual Merchandising turns out to be something of a beautiful textbook, with a history of window displays that in Olden Times (a.k.a. early 1900s) displayed not fashionable manikins, but rotating displays of art and sometimes curiosities (like a plane that had fallen while attempting to cross the Atlantic, and was promptly reassembled in a department store window).

With regards to a digital medium, the takeaways are a little harder to come by, though I guess you could force them out. For example, indoor arrangements depend on the store’s architecture, but there are a couple rules of thumb that help, such as “shelves that are angled rather than parallel to each other can draw people in to explore.” Could an interface be designed in such a way that the eye is drawn to viewing other goods for sale, rather than relying on a “Go to next page” click? How could you fashion an experience that allows someone to wander aimlessly, yet also not impede them if they want to get directly to Point B?

The three-dimensional space and atmosphere afforded by a store can be impactful in ways that I think are hard to mirror to video games, or even shopping websites. There’s the three-dimensionality, of course, and the impact of sound and other people and smells set to artificially make you feel like you’re in a warm tropical clothing paradise ripe for bikini buying. Also, even though I’m sure people are aware of the fact that stores try and push cheap gum and parcels of assorted high fructose corn syrup at you as you’re paying for your Actual Purchases, I think it’s easy to write off a lot of store design as “Oh, they did that just because that’s the way the store is shaped, not because they’re trying desperately to sink their hands into my wallet.” Contrast to perpetual social game pop-ups rewarding you with discounts on virtual goods.

Another thing that I believe stores will have going for them for a while is their ability to support social exploration, to serve as a bauble-filled backdrop to people’s wanderings as they share each other’s company. I can’t think of any games that do similar. (Though I suddenly imagine some WoW- or Second Life-esque scenario…ah, well, you could imagine an MMORPG fitting into any scenario, though.)

In short: Visual Merchandising has gorgeous photos, some interesting history and details about things like department stores and manikins, and so on. It also features some ~*famous window displays*~ like the one from Selfridge’s which has all those inflatable boobs…what does it look like? Read the book!
