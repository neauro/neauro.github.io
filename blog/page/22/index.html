Adding a New Model to a Database in Rails

Aug 16th, 2011

For whatever reason, adding a model to a database in Rails took me forever to understand. Thanks to Kelly, I now have model-making in Rails down to an art.

Want to add a new table or a new column to your Rails application? This is what you do!


1. Create a new .rb file for your model

Let’s say that, just like me, you want to create a new database table in your application to keep track of information about dogs. We (by virtue of the same logic which affords physics problems friction-less vacuums with which to work in) already have a couple tables in our database: one pertaining to users (breeders or owners of dogs), and one pertaining to breeds. We will keep it simple for now; all we want to know about a dog object for now is what its name is, what user it belongs to, and what breed it is.

To add a Dogs table, we first need to create a file which will describe its model. Part of this is describing what the relationship of the new model is to existing models, like User and Breed. From real-world experience, we know that a dog generally has one user/owner, and one breed.

So, create a file in app/models/ called dog.rb, and type in the following:
class Dog < ActiveRecord::Base
  belongs_to :user
  belongs_to :breed
end
Dog inherits Active Record. Active Record’s sole purpose of existence is to help create associations between different objects. This code will later be macro’ed out by Active Record into something more lower-level to create the associations.

The lines that start with belongs_to describe the relationship that this new model has to existing ones. Basically, a dog belongs to one owner/user, and to one breed.

Rails also has another keyword called has_one, which at first might also seem like an okay association to make between dogs and users; after all, a dog has_one user, right? But belongs_to is more correct, because has_one implies more of a one-to-one relationship, where you might be doing something like creating a new user every time you create a new dog. Since users may later obtain more dogs, belongs_to would be more appropriate in the case you want to attach a new dog to an existing user. (Note: belongs_to also makes it so that the Dogs table will have to be responsible for its foreign key with Users, whereas it would be the opposite case for has_one.)


2. Change your existing models to reflect their relationship to the new model

Because a User can have many dogs, and have many different breeds of dogs, User should now look like this: 
class User < ActiveRecord::Base
  acts_as_authentic
  has_many :dogs
  has_many :breeds, :through => :dogs
end
And because many dogs can be of a certain breed, and breeds can be owned by multiple owners, Breeds will look the same – 
class Breed < ActiveRecord::Base
  has_many :dogs
  has_many :users, :through => :dogs
end


3. Generate a new migration

Now you have to create a migration which, when run, will create the relevant tables in a structured manner. You could also be hipster about it and edit SQL on your database by hand, but then you’d would have to tell the other people you’re working with of what you’ve done if they have local databases, or else run the same SQL on production databases. With migrations, you can make changes modular, and even roll back changes easily.

Run rails g migration create_dogs_table. This will create a file in db/migrate which will look something like this:
20110807213821_create_dogs_table.rb
The numbers on the left are a timestamp, which is needed by Active Record so it will know what migrations that you haven’t yet run, so that it can run them for you.



4. Edit the migration file to describe what tables or columns should be added to your database.

Open your migration file and type in these things: 
class CreateDogTable < ActiveRecord::Migration
  def self.up
  end

  def self.down
  end
end
self.up is executed when a database is updated; self.down is executed when a database is rolled back due to errors. So, in self.up you want to create your new tables and columns; in self.down you want to define code to remove those tables and columns. With dog objects, all we said we cared about earlier was the name, so let’s just make a table which will (automatically) contain a unique ID, a timestamp, and a string for each dog object. 
class CreateDog < ActiveRecord::Migration
  def self.up
    create_table :dogs do | d |
      d.timestamps
      d.string :name
    end
  end

  def self.down
    drop_table :dogs
  end
end


5. Run the migration to create your table

Run the migration by typing rake db:migrate. Remember how the migration that we generated had a timestamp? When you run rake db:migrate, Active Record can use those to figure out all the migrations that might have been added to your code base, which have not yet been run on a certain database (such as the one pertaining to production). Also, because the code to create and drop data from a database is so high-level, it doesn’t really matter whether your actual code is in sqlite3 or in MySQL. You can even use both depending on whether you’re in development or production stages.

Hurray! Now you have new tables and columns in your database!

Editing Skype Chat Messages

Aug 10th, 2011

There is one thing in particular that I wish Skype would change about its UI, and I think it’s mostly influenced by my experience using another 2.0-y bright blue user interface.

I’m talking about the little side menu that comes up over your chat messages, which offers you a variety of possible actions for that message.

man, I should have taken a bigger screenshot

When I am clicking on my own chat, these are the things that I probably want to do, in order:

  1. Edit whatever foolish, silly thing I just wrote
  2. Remove whatever now-irrelevant thing I just wrote

I can only imagine Copy or Quote to be useful in cases when I want to be snarky.  (I already said that!  Exactly 5 minutes and 3 seconds ago!) Or maybe if I’m cross-posting between chats?  I don’t know.  I accept there is a use case that I haven’t encountered for those options.

Nevertheless, what I really wish I could do in Skype is click anywhere on a chat-message of mine and have it open up into something I could immediately edit, much like Player.IO’s treatment of objects in its database.  The menu could then be reserved for the lesser use cases of Removing, Copying, or Quoting.

Speaking of the menu, could we also make that little arrow larger?  The little arrow that opens that menu in the first place?

do you see it?

Since there’s nothing usually relevant to the right side of the chat-message anyway, it would help a lot if the hitbox for the menu was larger.  Also, if the menu opened up automatically, instead of me having to click it…!

Just thinking about it is exciting. :D

Good Design Moves Tumblr Has Made Recently Which I Really Appreciate

Aug 8th, 2011

Tiled Photosets
If you wanted to post more than one picture at a time on Tumblr, it used to be that the pictures would be placed into a little slideshow applet. It was really elegant and cute; you would see one picture at a time, and could press a little rounded arrow to move on to the next picture.

The problem is, no one would use it. Or at least, if people did post this way, it didn’t get interacted with. (At least, I didn’t, and the rare frequency with which I met up with a photoset on my dashboard makes me think that others didn’t really like this method of posting either.) On Tumblr people just scroll, scroll, scroll. Occasionally laugh. And don’t stop for your dang photosets.

Then, suddenly! A tiled photoset appears!

with varied photo sizes, even!

All of a sudden, you can see all the pictures in a photoset at once! Plus the given layouts for different pictures make them all look really artsy, which Tumblr users love, because let’s be honest, over half of us are there just reblogging stuff. Though another mark of how cool this feature is is how people have taken creative liberties with it, making sets of images like this, or this, or even this.

Making a feature which fits with how people actually use the product, and allows them to be creative? ++!


Reblogging Tags
I actually just noticed this today. Never again will I have to repopulate the tags of something that I’m reblogging. As an example, textless.tumblr is a great blog for macro photography of insects (and other animals), and the photographer always dutifully tags all of his posts with whatever is pictured. Wooly Aphid. Larva. Wasp Mimic. Secropia Silkmoth. (These tags often even get cut off in the dashboard post. I can’t find an example now, but may later.) In any case, though I love the accuracy of it, no way am I going to spend any time tagging these reblogs with anything more than “photo” and maybe “bug.”

Fortunately, now tags also get reblogged! So, now reblogging things can make you feel even more productive with even less energy. :)


thanks, tumblr.  you the bomb

RAWs vs. JPEG Fine

Jul 25th, 2011

I’ve done a lot of exploring in San Francisco this past weekend, as my more city-savvy friends have returned from Japan. One of them lamented having not known about raws before going, and we had a conversation about it, with me mostly being like, Dude, raw may make your pictures better but so does having the proper aperture, shutter speed, white balance, ISO, etc.

I’ve only tried shooting raw once, with somewhat non-exceptional results. The only thing that came out of this experience was that raws took forever to edit (on my poor, slow computer) and I didn’t really see much difference in quality.

Fast forward to exploration weekend, wherein I decided I would give raws another shot (if you will) and then realized that they are really cool. There are other places where you can learn exactly what raws are, so I won’t go into that here, but rather just talk about my experiences.

What I found really amazing about the raw images that I took was the capacity that I had to edit them. With my normal fine jpegs, I pretty much just threw them into Photoshop, and adjusted the middle bar of Levels slightly to the right to get rid of some of the grays.

now it’s less gray lol

And that was about the extent of my editing, unless I was trying to fix white balance or whatever. Occasionally I would touch the other two bars, but pictures would inevitably look way too bright or way too dark afterward. Plus, editing was pretty linear – stuff as a whole would just keep getting brighter (and way too bright), or else much darker.

now it’s more white lol

Enter raws, which are apparently just like having an undeveloped shot of film, in which all its possibilities of exposure are retained. Also enter the Conservatory of Flowers, which is what I did in order to take tons of pictures of flowers, despite the fact that flower picture-taking is uber-typical. Here is my original photo of a flower –


And here are two edited versions –





Where did all that detail come from?!

Had I been editing a jpeg from that original, I assure you that the only thing I would have known how to do is make that shit even whiter than it already was. Or else make everything around it as black as my soul. Though I have to admit that a likely cause of this is just the fact that using Photoshop tools is like reaching for something in a huge, bubbling cauldron and slapping it on your picture – you don’t really know what you’re doing and you’re never really quite sure what will happen.

what wait oh no why

On the other hand, the workflow for editing a raw in the Mac that I’m borrowing is pretty self-explanatory.

oh, so those are the things i can do?
and i can see what changes they make to the picture immediately?
okay.

Anyway. My notable observations on raw are as follows:

  • Raw files take up too much dang space.  My 4GB memory card can handle 1,500+ fine JPEGs like a badass.  It can hold about 400 raws.
  • This means that my normal methods of “TAKE AS MANY PICTURES AS YOU CAN AND MAYBE THREE OF THEM WILL BE OKAY” won’t work here.
  • Then again, the reason why I had to take so many pictures was because so many of them turned out irreversibly poor.  And if I could edit them…
  • Even though it sucks to have less space, I don’t actually mind it – throughout this past weekend I found myself being more thoughtful about what pictures to take, and being more vicious about which ones to immediately delete if I didn’t like them.  It’s not just memory that I need to take into account – it’s the fact that every picture I save will require some minutes of editing, which, in my busy busy day is worth   a ton.  I’m already two days behind in my San Francisco Summer thing I’m trying to do.
  • Being more thoughtful in the actual picture-taking process is, to me, a refreshing change.
  • Plus, in the actual editing I really feel more connected and appreciative of each picture, and thus more like I’ve actually accomplished something meaningful for myself, etc.

That being said, aside from pictures which will be explicitly harvested for panorama purposes, I think I’m going to do my best to stick to raw file formats as a default from now on, the same way that I moved somewhat mysteriously from always-auto to always-manual settings.  Still, I think it’s important to have a handle on aperture/shutter speed/ISO and all that, since a raw file format can’t fix blur.  (That I know of.)  And you know me, I’m liek totally super pro photog, so listen up.
